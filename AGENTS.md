# AGENTS.md

## 1.概要

このドキュメントは、Streamlit内でCloudFlare D1を用いた試作プロダクトを開発するにあたり必要なルールやツールセット、基本的なビルドやテスト実行のコマンドやAIエージェント向けの要請・許可・禁止事項などについて定めたものです。

## 2.プロジェクトの目的

本プロジェクトは、Streamlit上で動作するCloudFlare D1の試作品開発を通じてボイラープレートとなりうるプロジェクトのファイルセット作成することを目的とします。

### 3.開発環境・コーディング規約

* **Pythonバージョン**: 3.13
* **Pythonインタプリタ自体の管理ツール**: mise
* **依存関係管理ツール**: uv
* **コーディング規約**: PEP 8準拠
  * importについて
    * import文は `from X import Y` 構文を使わず、絶対 import を使用する。例: `import package.module` の形式を用いる。相対 import も禁止する。
    * ただし自パッケージ内のモジュールについては相対importを許可する。
  * タイプヒンティングについて
    * 可読性とメンテナンス性を高めるため、タイプヒントは可能な限り記述
    * ただし定数の代入や関数の戻り値型が一意であるなど右辺から推論が自明な場合は省略
* **フォーマッター/リンター**: ruff
* **単体テストフレームワーク**: pytest

ディレクトリ構造
* いわゆるsrcレイアウトを採用している
  * 成果物は基本src/ディレクトリ以下のパッケージ内に作成を想定している
* テストコードはtests/ディレクトリ以下に配置する

## 4.開発ワークフローとコマンド

全ての開発ツールとスクリプトは、プロジェクトの仮想環境で実行するため常に `uv run <コマンド>` 形式で実行すること（例: `uv run pytest`, `uv run ruff format`）。`uv` が利用できない場合は、まずユーザーに「uv が見つかりません。プロジェクトで uv をインストールしてもよいですか？ (はい/いいえ)」と許可を求め、ユーザーが『はい』と答えた場合にのみ `mise use uv` を実行して続行する。

未インストールのツールを検知した場合は、(1) 必要なツールの一覧を提示し、(2) ユーザーからの個別承認が得られたツールだけをインストールすること。ユーザーが明示的に「すべて許可」とした場合のみ一括インストールを行ってよい。

下記にコマンド例を示す。
* **パッケージ追加**: `uv add <package>`
* **単体テスト実行**: `uv run pytest`
* **コードチェック実施＆修正**: `uv run ruff check --select I --fix`
* **コードフォーマット実施**: `uv run ruff format`

コマンドが失敗した場合は、以下の手順で対処すること。
1) 実行結果はプロジェクトルートの `.logs/` ディレクトリに保存する。ファイル名は `<command>-<YYYYMMDD-HHMMSS>.log` とする。ログには少なくとも以下を含めること: 実行コマンド、実行時に公開可能な環境変数、`stdout`、`stderr`、終了コード。
2) 同じコマンドを1回だけ自動で再試行し、再試行結果は同一ログファイルに追記すること。
3) 再試行後も失敗する場合は、次のテンプレートでユーザーへ報告する: `実行コマンド: <cmd>, 失敗理由: <stderr（要約）>, ログ: <project_root>/.logs/<file>`。報告とともにログファイルのパスを添えてユーザーの指示を仰ぐこと。

### 5.テスト方針
テスト方針として以下を文書化すること：
1) 目的・対象範囲・テストレベル
2) ローカルおよびCIでの実行手順
3) カバレッジ測定方法と使用ツール
4) カバレッジが80%未満だった場合の処置手順

* **テストフレームワーク**: pytest
* **テストカバレッジ目標**: 行カバレッジ(line coverage)で80%以上を目指す（ツール: `pytest --cov` を使用）。ブランチカバレッジ要件が必要なら別途指定する。
* **CI (継続的インテグレーション)**: 未定。CIが未定の場合は自動でCIを作成しないこと。CI導入が必要な場合は候補（例: GitHub Actions）を提案し、ユーザー承認を得てから設定を行う。
* カバレッジが80%未満の場合、`pytest --cov` を実行して未カバー箇所の一覧を生成し、優先度の高い未カバー関数/モジュールのテスト追加を提案する。テスト追加にユーザー許可が必要なら問い合わせること。

### 6.要請・許可・禁止・補足

次の事項は常に守ること。
* 応答やImplementation Plan、Walkthroughはすべて日本語で作成
* コード作成・変更後にはリンタやフォーマッタ実行を実行する
* 自身が加えた変更がさらに人間の手によって変更される可能性もあることを意識する
* 指示された事以外の改善などは提案はしてもよいが、許可を得ずに変更まで加えない
* 全角の括弧は使用しない

次のことは許可を求めずにやってよいものとする。
* プロジェクトディレクトリ配下のファイルの読み取り・書き込み操作
* リンタやフォーマッタ実行
* インターネットアクセスは、パブリックに公開されたドメイン（例: `docs.python.org`, `pypi.org`, 公式APIエンドポイント）へのドキュメント参照および未認証の公開APIリクエストのみ許可する。認証が必要なAPIや社内/非公開ドメインへのアクセス、及びプロジェクトディレクトリ内へのファイルダウンロードは事前にユーザー許可を得ること。プロジェクト外の一時ディレクトリへのダウンロードは、ユーザーの事前許可があれば可とする。

次のことは許可なしには実行してはならない。必要であれば許可を求めること。
* タスク遂行や課題解決の上で必要と思しき依存関係の追加
* インターネット環境からプロジェクトディレクトリ内へファイルダウンロード

| action type | permission required | allowed | default behavior |
| --- | --- | --- | --- |
| 依存関係の追加 | はい | いいえ | ユーザー許可を得るまで中止 |
| プロジェクト内ファイル操作 | いいえ | はい | 通常実行 |
| リンタ/フォーマッタ実行 | いいえ | はい | 自動実行可 |
| プロジェクトへのファイルダウンロード | はい | いいえ | ユーザー許可を得るまで中止 |
| 公開APIやドキュメント参照 | いいえ | はい | 目的に応じて実行 |

次のことは一切禁止とする。
優先順位と判断フロー: 操作を実行する前に以下の順で判断すること。
1) 操作が明示的に禁止されていないか確認する（明示的禁止が最優先）。
2) テーブルの 'permission required' 列を参照し、`はい` の場合はユーザー許可を求める。`いいえ` の場合は許可不要として進める。
3) 複数のルールが衝突する場合は、上位のルール（明示的禁止 > permission required: はい > 許可不要リスト）の順で解決する。
* プロジェクトディレクトリよりも上の階層のファイルやディレクトリに対する一切の読み取り・書き込み操作
* 機密情報のハードコーディング
  * 機密情報は `.env` ファイルまたはシークレットマネージャに保存する。エージェントは `.env.example` テンプレートを作成できるが、実際のシークレット値を含むファイルをリポジトリに追加する前にユーザー許可を得ること。
* gitリポジトリへのpush操作

また上記に該当せず、且つ要件が曖昧な場合や不明確な場合は、推測で変更を行わず必ずユーザーに質問すること。要件確認の際は次のテンプレートを使用すること:

"確認: <具体的に不明な点>。想定される選択肢: A) <...>, B) <...>. どちらを採用しますか？"

リポジトリ内で疑わしいシークレットが検出された場合の手順:
1) 自動でコミットやファイルを書き換えないこと。
2) 検出したファイルパスと検出内容、推奨対応（例: 値を環境変数へ移動、該当シークレットのローテーション）を一覧化してユーザーに報告すること。
3) ユーザーが明示的な許可を与えた場合にのみ、指定された対応（ファイルの削除や置換、.env.example の生成など）を実行すること。
